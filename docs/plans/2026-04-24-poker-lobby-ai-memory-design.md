# Texas Poker CLI 大厅模式与 AI 灵魂记忆设计

日期：2026-04-24

## 背景

当前项目已经具备以下能力：

- 通过 `server` / `client` / `slacker` / `ai` 命令启动多人德州扑克
- 使用 Socket.io 管理房间与状态同步
- 使用 blessed 渲染普通 CLI 界面与摸鱼模式界面
- 使用大模型驱动 AI 玩家行动

当前痛点：

1. 启动流程偏工程化，普通玩家需要记很多命令
2. AI 决策只看当前牌局上下文，不会形成“人格”和“本次会话记忆”
3. 玩家无法把 AI 当作有个性的牌桌角色来观察和博弈

本设计目标是：

- 把默认入口改成大厅 / 主菜单模式
- 支持从大厅进入本机局或联机局
- 为每个 AI 建立本次会话有效的 `soul + memory`
- 允许 GM 与普通玩家看到不同粒度的玩家画像

## 总体产品方案

默认启动进入大厅。

大厅第一层提供两个入口：

- 本机局
- 联机局

### 本机局流程

玩家在大厅中配置：

- 真人名称
- 总人数
- AI 数量
- UI 模式（普通 / 摸鱼）

确认后系统执行：

1. 自动检测或启动本地服务器
2. 自动创建房间
3. 自动生成并加入 AI 玩家
4. 自动让真人进入牌桌 UI

用户日常只需要一个入口，例如 `npm run play`。

### 联机局流程

大厅中提供：

- 创建房间
- 加入房间

创建房间后，房主可在开局前补充 AI 数量并开始游戏。

### 兼容策略

保留现有 `server` / `client` / `slacker` / `ai` 命令，作为高级入口与兼容入口，不破坏现有使用方式。

## 架构设计

推荐采用“主菜单驱动 + 会话态中心化”方案。

### 新增模块

#### 1. `LobbyRenderer`

负责渲染大厅 / 主菜单 UI，收集启动配置，不直接参与牌局规则。

职责：

- 展示大厅菜单
- 收集本机局 / 联机局配置
- 输出统一的 `GameLaunchConfig`

#### 2. `GameLauncher`

负责根据 `GameLaunchConfig` 执行启动流程。

职责：

- 本机局：启动 server、建房、拉起 AI、启动 UI
- 联机局：连接已有 server、创建房间或加入房间
- 统一管理启动期的错误处理和回退提示

#### 3. `SessionManager`

这是本设计的核心新增层，用于维护“本次游戏会话”的整体上下文。

职责：

- 记录本次启动以来的所有对局历史
- 聚合每手牌的行动、耗时、输赢
- 维护玩家公开画像和 AI 内部画像所需数据
- 为 AI prompt 生成摘要输入

#### 4. `AiMemoryStore`

由 `SessionManager` 持有，为每个 AI 保存本次会话有效的短期记忆。

职责：

- 存储 AI 的 `soul`
- 存储 AI 的会话记忆
- 存储 AI 对其他玩家的观察与判断摘要

#### 5. `ProfileVisibilityService`

负责按视角生成玩家简介。

职责：

- 生成 GM 可见的完整档案
- 生成普通玩家可见的逐步揭示版档案

## 数据模型设计

### `GameLaunchConfig`

建议字段：

- `mode`: `local` | `online`
- `playerName`
- `roomId`
- `host`
- `uiMode`: `cli` | `slacker`
- `totalPlayers`
- `aiCount`
- `isGm`

### `SoulProfile`

每个 AI 固定拥有一份 `soulProfile`。

生成方式采用“原型 + 随机扰动”。

建议首批原型：

- 数学派
- 赌徒派
- 表演派
- 复仇派
- 老狐狸
- 胆怯派

建议维度：

- `riskTolerance`
- `discipline`
- `aggression`
- `ego`
- `vengefulness`
- `patience`
- `deception`
- `emotionalVolatility`

这些维度不直接硬编码行动逻辑，而是作为人格底色进入 prompt。

### `AiSessionMemory`

每个 AI 在本次会话中维护以下信息：

- 自己的 `soulProfile`
- 当前手牌上下文摘要
- 本手行为历史
- 跨手关键事件
- 各玩家观察摘要
- 自己的近期输赢轨迹
- 自己对桌面气氛和对手的主观判断

### `PlayerObservation`

用于描述某个 AI 眼中的其他玩家。

建议内容：

- 对方最近几手的下注模式
- 常见行动速度
- 是否频繁施压
- 是否偏被动
- 是否对自己有针对性
- 可供模型自由解释的开放式备注

### `HandHistoryEvent`

建议每个动作都记录为结构化事件。

字段包括：

- `handId`
- `phase`
- `playerId`
- `action`
- `betBefore`
- `betAfter`
- `raiseTo`
- `potAfter`
- `thinkTimeMs`
- `timestamp`

### `SessionOutcome`

每手结束时记录：

- 胜者
- 摊牌信息
- 每个玩家本手盈亏
- 关键事件标签

## AI 设计：Soul + Memory，而非硬编码情绪

本设计不在程序里直接判断“某 AI 现在 tilt 了”“某 AI 现在自信了”。

相反，系统只负责尽可能完整地记录场上信息，并交给模型结合自身 `soul` 自然推理。

AI 决策时输入由三部分构成：

- `soul`：它是谁
- `memory`：这次游戏里经历了什么
- `state`：当前这一手牌是什么局面

这样 AI 可以自然产生下列风格差异：

- 因连输而保守
- 因被某人压制而记仇
- 因连赢而更加高压
- 因人格偏表演而故意制造误导

目标不是做“最优解 AI”，而是做“有性格、会演化、可观察”的牌桌对手。

## Prompt 设计

### Prompt 输入层次

每次 AI 决策时，建议生成以下结构：

#### 1. 角色自我定义

- 名字
- `soul` 原型
- 主要人格维度
- 简洁的人格叙述文本

#### 2. 当前牌局信息

- 手牌
- 公共牌
- 当前阶段
- 底池
- 当前下注
- 自己需要补多少
- 各玩家筹码与状态

#### 3. 本手行动历史

- 谁在什么阶段做了什么动作
- 动作的下注量
- 行动耗时

#### 4. 本次会话摘要

- 最近若干手的输赢
- 最近关键冲突
- 对主要玩家的观察笔记
- 自己最近的波动与倾向

#### 5. 行为约束

输出只允许：

- `fold`
- `check`
- `call`
- `raise <amount>`

### Prompt 原则

- 不把完整原始日志无脑塞给模型
- 由 `SessionManager` 先做摘要，再给模型
- 既保留细节，又控制 token 成本

## 玩家简介 / 档案卡设计

### 功能目标

让 AI 从“后台参数”变成“牌桌角色”。

### 视角差异

#### GM 视角

可查看完整信息：

- `soul` 原型与维度
- 会话记忆摘要
- 近期关键事件
- 当前内部判断摘要

#### 普通玩家视角

不能看到真实 `soul`，只能看到逐步揭示的推测画像。

普通玩家可见简介的形成依据：

- 已观察到的下注风格
- 已见过的摊牌
- 与该玩家交手的历史
- 最近公开发生的重要事件

### 文案风格

普通玩家看到的是叙述式描述，而非内部数值。例如：

- “这名玩家出手不多，但关键时刻很敢压。”
- “最近输了几手后，他的动作开始更急。”
- “他似乎对某位对手格外在意。”

GM 则可以额外看到结构化信息。

## 服务端事件采集设计

为了支持会话记忆，需要补充现有行动采集能力。

建议在客户端与服务端之间加入以下数据采集：

- 行动开始可操作时间
- 玩家真正提交动作的时间
- 由此计算 `thinkTimeMs`

采集策略建议：

1. 服务端广播当前轮到谁行动及行动开始时间
2. 客户端提交动作时携带本地动作时间戳
3. 服务端以自身时间为准，必要时融合客户端时间做近似修正

需要保证：

- 即使是 AI 自动行动，也有真实可记录的思考延时
- 耗时进入 `HandHistoryEvent`

## 与现有代码的集成策略

### 尽量不重写 `Game`

`Game` 继续负责：

- 发牌
- 盲注
- 回合流转
- 阶段推进
- 胜负判断

### 让外层系统记录更多语义信息

新增的行为分析、记忆、画像、prompt 生成尽量放在 `Game` 外层。

推荐在 `Server` 周边增加一层事件流水记录，而不是让 `Game` 直接承担这些职责。

## 第一阶段实施范围

建议第一阶段只做最有价值的闭环：

1. 新增大厅入口，默认通过 `play` 进入大厅
2. 支持本机局 / 联机局菜单选择
3. 支持本机局自动配置 AI 并启动
4. 建立 `SessionManager` 基础结构
5. 为 AI 增加 `soulProfile`
6. 记录本手动作、下注额、行动耗时
7. 为 AI prompt 增加 `soul + recent session summary`
8. 新增 GM/普通玩家两种简介视图的基础版本

## 后续扩展方向

第一阶段完成后，可继续扩展：

- 更丰富的原型库
- AI 之间的相互偏见和关系记忆
- 手牌结束后的“新闻播报”或“牌桌旁白”
- 更高级的普通玩家画像揭示机制
- 面向 GM 的后台控制面板

## 测试策略

建议测试分三层：

### 1. 单元测试

- `SoulProfile` 生成
- `SessionManager` 汇总逻辑
- `ProfileVisibilityService` 视图裁剪逻辑
- prompt 摘要生成逻辑

### 2. 集成测试

- 本机局大厅启动流程
- 自动补 AI 后进入游戏
- 一手结束后记忆更新

### 3. 人工试玩验证

重点验证：

- 启动是否足够简单
- AI 是否表现出稳定人格差异
- 简介是否真的“像一个玩家”
- GM / 普通玩家的可见边界是否合理

## 风险与注意事项

- 当前代码的房间与牌局逻辑较轻量，事件采集接入时要避免把核心状态机耦合得过重
- AI prompt 如果摘要不好，容易只变成长 prompt 而不变得更聪明
- 玩家简介若泄露过多内部信息，会损害博弈感
- 普通玩家视角必须坚持“推测画像”，不能等同后台真相

## 结论

本次设计将项目从“可玩的 CLI 德州扑克”升级为“带大厅、带人格 AI、带会话记忆、带观察乐趣的扑克对战系统”。

核心原则只有三条：

1. 默认入口简单，用户只要会进大厅即可
2. AI 不是规则脚本，而是拥有 `soul + memory` 的牌桌角色
3. 信息可见性分层，GM 看真相，普通玩家靠观察逐步读人
